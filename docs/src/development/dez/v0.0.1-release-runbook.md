# Dez v0.0.1 release runbook

This is the operator-facing release record for the first Dez preview. It
collects release notes, recovery behavior, known limitations, artifact identity,
checksums, and rollback guidance. Direct command and runtime proof remains in
[Release Evidence](./release-evidence.md); permanent product decisions remain in
[Fork Notes](./fork-notes.md).

## Release state {#release-state}

Dez v0.0.1 is **not a public release yet**. The local arm64 development bundle
is built, ad-hoc signed, structurally audited, and exercised against protocol 4.
The following gates remain open:

- corrected visual-state and narrow-width capture;
- macOS accessibility tree, focus order, and keyboard/pointer parity;
- graceful GUI quit, same-Session reattach, and same-child-PID proof;
- structured Codex attention/review/restart hero flow;
- official Zed installed coexistence;
- app-facing modified-crate Clippy;
- Developer ID signing, notarization, install/uninstall, and updater audit;
- design-partner validation on real repositories.

An unchecked gate is not implied by source, compilation, or an adjacent test.

## What is in the v0.0.1 source candidate {#release-notes}

- A terminal-native Session Rail that projects ordinary shells, terminal
  agents, and native agents without making detection decide whether a terminal
  exists.
- All/Attention scopes, visible search, deterministic selection, caught-up,
  no-result, empty-workspace, missing, incompatible, reconnecting, detached,
  exited, and failed states.
- An opt-in authenticated local terminal Host with stable Host/Session IDs,
  bounded replay, input, resize, heartbeat, event cursors, server-pushed
  snapshots, detach, explicit terminate, and observed-exit truth.
- Protocol 4 replay geometry: snapshots and replay fragments carry their
  recorded columns and rows so restored output is not spatially reinterpreted
  at one arbitrary terminal size.
- Conservative structured Codex lifecycle, command, check, permission,
  attention, file-target, and review evidence. Missing evidence remains visible
  instead of being guessed.
- A deterministic editable Review Brief beside the owning surface, with
  observed Git/worktree state, bounded changed paths, checks, risks, and review
  outcomes.
- A quiet local-first shell with upstream account, subscription, promotion,
  collaboration, support, social, and hiring chrome removed or explicitly
  labeled as upstream compatibility. Automatic Zed websocket/LiveKit startup
  and inherited Zed-hosted prediction providers are also gated in source;
  explicit non-Zed prediction providers remain available.
- A repaired macOS client-decoration layout, one mode-resolved reservation and
  paint width for the Session Rail, a restored-project launch surface that
  cannot degrade to a blank center, deliberate empty states, a single-line
  truncating workspace footer with bounded identity/Git regions, and a named
  accessibility landmark.

## Terminal recovery contract {#terminal-recovery}

For an eligible ordinary local shell when the experimental Host is enabled:

1. Creating a terminal creates one Host-owned Session and one child process.
2. Closing its surface detaches the view; it does not terminate the Session.
3. A detached Session remains in Session Rail with its Host/Session identity.
4. Reopening attaches to that Session, replays only bounded retained output,
   restores recorded terminal geometry, and continues from the current cursor.
5. Exiting the shell records observed `Exited` state.
6. `Stop` is an explicit destructive Host request and stays separate from
   close, detach, archive, and delete.
7. If the Host is missing, incompatible, malformed, or unavailable, Dez opens
   a display-only recovery surface. It does not silently start replacement
   computation or rerun an agent initialization command.

Task terminals remain GUI-owned in v0.0.1. Their cancellation, rerun, and
completion semantics make automatic cross-GUI survival unsafe; use an ordinary
durable shell for long-running supervised work.

### State meanings {#state-meanings}

| State          | Meaning                                                                         | Safe next action                                                 |
| -------------- | ------------------------------------------------------------------------------- | ---------------------------------------------------------------- |
| `Live`         | The Host observes a running Session.                                            | Focus or attach.                                                 |
| `Detached`     | The Host owns the Session but no GUI surface is attached.                       | Reopen it from Session Rail.                                     |
| `Reconnecting` | The GUI has not re-established authoritative Host state.                        | Wait, copy diagnostics, then restart Dez if it persists.         |
| `Exited`       | The Host observed process exit.                                                 | Review retained output or start a new terminal explicitly.       |
| `Missing`      | Saved identity exists, but the expected Session does not.                       | Inspect recovery details; start new work only deliberately.      |
| `Incompatible` | Host protocol/capability negotiation cannot represent the saved Session safely. | Update/restart the matching Dez components; do not force replay. |

## Known limitations {#known-limitations}

- Durable local shells are still behind
  `DEZ_EXPERIMENTAL_TERMINAL_HOST=1`; the helper is not yet the default backend.
- The accepted v0.0.1 storage compatibility boundary still uses existing
  `Superzed` user-data directories. This prevents apparent loss of settings,
  extensions, history, and local agent state until a transactional migration
  exists. It is not public product branding.
- Codex hook installation is manual and Unix-shell-oriented. Dez never edits
  the user's Codex configuration automatically.
- Terminal replay is bounded. Truncation is explicit; Dez does not claim to
  preserve unlimited scrollback.
- Generic process/title detection is lower-confidence classification, not
  provider lifecycle truth.
- Review Brief outcomes are editable review notes; they do not mutate or resume
  the owning Run.
- Repository-wide Git changes cannot be attributed to one agent without direct
  evidence.
- Public signing, notarization, updater ownership, installed coexistence, and
  migration UX are not proven by the local ad-hoc bundle.
- The current exact bundle predates any later unbuilt source edits. Rebuild,
  re-sign, and re-audit before using a changed source tree for release evidence.
- In particular, the current exact bundle predates `79f69b273c`, which fixes
  the compact rail painting 300 px of content into a 240 px allocation. Do not
  use the current bundle as visual evidence for that correction.
- It also predates `4829f6b052`, which restores Project ready actions when a
  loaded legacy/restored pane lacks its welcome flag. The current bundle is not
  visual evidence for the blank-center correction either.
- It predates `1d5c03d88b` and `9318b270d9`, which enforce and statically guard
  the local-first startup boundary. Do not use the current bundle as evidence
  for quiet/offline launch behavior.
- It predates `0d8496969f`, which gives project identity and worktree/branch
  controls bounded shrink regions in the lower workspace footer. Do not use the
  current bundle as evidence for narrow-width footer behavior.

## Safe local verification {#safe-local-verification}

Before a runtime gate:

1. Confirm the intended bundle path and identity.
2. Confirm deep strict signature verification.
3. Confirm no unintended Dez/Superzed binary is running.
4. Freeze the source commit, lockfile hash, executable hashes, and test scope.
5. Launch only the exact candidate. Do not use the historical `dist` bundle to
   work around accessibility or registration problems.

For a recovery gate, record the GUI PID, helper PID, Host ID, Session ID, child
PID, replay cursor, dimensions, retained markers, and lifecycle state before
and after a graceful application quit. A signal delivered through a shared
foreground PTY is not valid GUI-lifecycle evidence because it can also reach
the helper or shell process group.

## Rollback {#rollback}

Rollback must preserve user work and distinguish the GUI from Host-owned
computation:

1. Save editor changes and note every live Host/Session ID.
2. Quit the exact Dez GUI normally. Do not terminate the helper merely to remove
   the window.
3. If an installed preview exists, move only that exact `Dez Dev.app` or
   `Dez.app` bundle to Trash. Do not delete Zed, Superzed compatibility data, or
   shared user directories.
4. Restore source changes with reviewable `git revert` commits. Do not reset the
   repository or discard unrelated working-tree changes.
5. Rebuild the previous frozen commit and re-run its identity/signature audit.
6. Terminate a Host Session only when the user explicitly intends to stop that
   computation. Record the termination response before removing metadata.

The local Cargo dependency cache and generated test executables are disposable
and can be downloaded or rebuilt. Removing them does not roll back application
state, settings, terminal metadata, or user files.

## Public preview promotion gate {#promotion-gate}

Promote this candidate only when every open item in Release Evidence is checked
with direct proof and the authoritative repository `PLAN.md` has no v0.0.1
release blocker. The local ad-hoc signature must be replaced with the Dez
publisher identity and notarized artifact, and installed coexistence must show
that Dez cannot overwrite or update as official Zed.
