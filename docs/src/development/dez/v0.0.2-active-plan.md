# Dez v0.0.2 active plan

This document is the active release train for Dez v0.0.2. It turns the
permanent decisions in [Fork Notes](./fork-notes.md) into ordered work and
release gates. It does not override Fork Notes, and it should be replaced when
v0.0.2 ships.

## Release promise

Dez v0.0.2 helps a terminal-native developer supervise a long-running coding
agent without losing the terminal that owns the process or the editor context
needed to review its work.

The release-defining workflow is:

1. Open a project.
2. Start a hosted terminal session and run an agent or another long-lived
   command.
3. Move between terminal, editor, search, Git, and agent review without losing
   session identity.
4. Close and reopen the Dez GUI.
5. Reconnect to the same hosted process and terminal history.
6. Inspect changed files and diffs in the editor.
7. Recover safely when the host, shell, project, or credentials are unavailable.

The IDE is not a separate product beside the terminal supervisor. It is the
inspection and intervention surface around the durable terminal: files,
language intelligence, search, Git, diffs, diagnostics, and agent context all
operate on the same project and session.

## Evidence vocabulary

Every status claim must use one of these labels:

- **Source**: implemented and inspected in source.
- **Compiled**: the relevant target compiled.
- **Runtime**: exercised in the current Dez binary with recorded observations.
- **Released**: present in a signed, checksummed, downloadable artifact.

Passing a lower evidence level never implies a higher one.

## Ordered release gates

### Gate 0: upstream parity and identity

- Complete a two-parent merge of the recorded Zed target.
- Keep `Dez`, `dez`, the Dez bundle identifiers, updater identity, and terminal
  host protocol distinct from Zed.
- Record every conflict and deliberate divergence.
- Make future upstream synchronization repeatable.

### Gate 1: reproducible build

- Build `dez` and `dez-terminal-host` from the locked workspace.
- Run formatting, identity, shell syntax, metadata, and diff checks.
- Record artifact sizes, types, and SHA-256 checksums without launching another
  binary.

### Gate 2: durable terminal contract

- Starting a hosted terminal creates one visible session with a stable identity.
- Closing the GUI detaches instead of terminating its hosted process.
- Reopening the GUI reconnects to the same process and restores useful history.
- Explicit stop terminates the hosted process group and produces an honest final
  state.
- Ordinary terminals retain upstream close and process-group cleanup behavior.

### Gate 3: coherent IDE integration

- Opening a session selects the related terminal and project context.
- Editor, search, Git, diagnostics, and diffs remain normal first-class IDE
  tools.
- Agent output can lead to the exact changed file or diff without creating a
  second navigation model.
- Pane and panel actions use predictable focus, close, split, and restore rules.

### Gate 4: failure and recovery

- Missing host, disconnected host, exited process, invalid working directory,
  and authentication failure each have distinct copy and recovery actions.
- Retry is safe and never creates an invisible duplicate process.
- Destructive actions identify the target and state what will happen.
- Empty and loading states never resemble a broken blank window.

### Gate 5: visual, input, and accessibility quality

The following matrix must be inspected at compact, default, and wide window
sizes:

| Surface         | Required states                                                 |
| --------------- | --------------------------------------------------------------- |
| Session rail    | empty, populated, attention, disconnected, search, overflow     |
| Terminal thread | running, idle, command active, exited, detached, reconnecting   |
| Editor          | file, diff, diagnostics, preview tab, split                     |
| Agent review    | streaming, complete, tool activity, error, changed files        |
| Recovery        | missing host, stale session, invalid project, retry in progress |

Each state must be usable with keyboard and pointer, expose visible focus, retain
legible contrast in Lumin Light and Lumin Blur, respect reduced motion, and
present meaningful labels to assistive technology. JetBrains Mono remains the
terminal and code default where available; UI typography must use the design
tokens rather than ad hoc font sizes.

### Gate 6: security and trust

- Hosted-process ownership and cleanup are explicit.
- Secrets are not written to session metadata, logs, or recovery copy.
- Agent and terminal actions reveal their working directory and trust boundary.
- Risky actions require a clear confirmation appropriate to their impact.
- Any upstream sandboxing change is audited and documented before release.

### Gate 7: preview release

- Package and sign the macOS Apple-silicon application and helper.
- Verify clean install, upgrade, relaunch, reconnect, and removal paths.
- Publish checksums, known limitations, recovery semantics, and rollback steps.
- Promote only after the hero workflow passes from a clean installation.

## Next ten tasks

1. Commit the two-parent upstream merge and update synchronization provenance.
2. Audit the upstream sandboxing change and choose an explicit v0.0.2 treatment.
3. Exercise attach, GUI close, reopen, reconnect, explicit stop, and process-group
   cleanup against the built artifacts.
4. Repair the terminal-thread hierarchy, spacing, status language, and recovery
   states using the visual matrix above.
5. Verify editor, diff, Git, search, diagnostics, and agent navigation as one
   project workflow.
6. Run keyboard, focus, screen-reader-label, reduced-motion, Lumin Light, and
   Lumin Blur acceptance checks.
7. Verify a clean-install host bootstrap and explicit recovery from missing or
   stale hosted sessions.
8. Verify no Zed updater, endpoint, storage, bundle, or CLI identity can be
   selected by Dez.
9. Package and sign the macOS Apple-silicon preview.
10. Complete release evidence, known limitations, rollback instructions, and
    the v0.0.2 runbook.

## Explicitly outside v0.0.2

- A speculative split into new orchestration, event, policy, scheduler, adapter,
  artifact, CLI, or headless-daemon crates.
- Remote execution, shared cloud sessions, mobile or browser clients, and team
  administration.
- A second editor, terminal emulator, Git client, task system, or extension
  platform alongside Zed's existing subsystems.
- Claims of reliability, accessibility, security, or performance that lack the
  evidence level stated above.
